# RandomGen — 360° Code Review, Structural Audit & Analysis

- Date: 2026-06-24
- Subject: `randomgen` v0.19.0 (API contract 2.1.0), branch `main`
- Method: multi-agent review — 10 dimension reviewers fanned out over the
  codebase, every finding adversarially re-verified against its cited
  `file:line`, then synthesised. 48 agents, 37 findings raised, 33
  survived verification (4 dropped), deduped to 29 reported findings.
- Scope guard: every recommendation was reconciled against arc42 §3.3.
  Deliberate exclusions (auth, rate limiting/CORS, crypto-RNG, persistence,
  TLS) are recorded under [Scope-excluded by design](#scope-excluded-by-design),
  not raised as gaps.
- Status: read-only assessment. No source was changed. Actionable findings
  are grouped into the GitHub issues listed under [Issue map](#issue-map).

## Overall: A−

RandomGen is a genuinely strong portfolio/demo. The architecture, container,
dependency, and API-contract dimensions are exemplary for a project this
size: a framework-free `domain/` that honours the Clean Architecture
dependency rule in practice (not just in prose), a textbook `create_app()`
composition root, a version registry that decouples routes from generators,
a faithful design-first OpenAPI contract with automated drift guards, a
one-gate-per-job CI/CD pipeline with SHA-pinned actions and least-privilege
tokens, and a digest-pinned non-root Alpine image. The ADRs scope the design
honestly rather than cargo-culting ceremony.

The one material defect is a correctness/serialisation bug: a valid
single-category distribution (e.g. `dist=5:1.0`) yields `df=0`, a `NaN`
p_value, and an HTTP 200 body containing the bare token `NaN` — not valid
RFC 8259 JSON, and a violation of the `type: number` contract in
`openapi.yaml`. It has zero test coverage and was reproduced live during
this review. Everything else is low/info hygiene: validation duplicated five
times, three dead `__main__` demo blocks, a performance test whose bound is
10× looser than its name, missing security response headers, an unbounded
distribution-category count, and several supply-chain hardening gaps that the
project's own devsecops template calls for.

Recommendation: fix the `NaN` edge and its test first, then work the cleanup
backlog.

## High severity

### F1 — Single-category distribution yields `NaN` p_value as invalid JSON in a 200 response (untested)

Category: correctness · Files: `domain/hypothesis.py`, `service.py`,
`blueprints/api.py`, `tests/integration/test_api_contract.py` · Effort: M ·
Independently reproduced during this review.

A single-category distribution is valid input: `validate_distribution`
accepts `values=[5], probabilities=[1.0]`. In `ChiSquareTest.calc`,
`self.df = len(contributing) - 1` (hypothesis.py:315) is 0, so
`self.p_value = 1 - chi2.cdf(self.chi_square, self.df)` (hypothesis.py:318)
becomes `1 - chi2.cdf(0.0, 0)` = `NaN`.

Live reproduction: `GET /api/v1/randomgen?numbers=10&dist=5:1.0` →

```
HTTP 200
{"numbers":[5.0,...],"quality":{"chi_square_test":{"chi_square":0.0,"df":0,"is_null":false,"p_value":NaN},...}}
```

`openapi.yaml:162-163` declares `p_value` as `type: number`. The bare token
`NaN` is not legal JSON (RFC 8259); a standards-compliant `JSON.parse` throws.
The verdict is also wrong: `is_null:false` reports rejection of the null
hypothesis although the sample perfectly matches the expected distribution.
No test exercises a single-category distribution.

Fix: guard the degenerate domain in `ChiSquareTest.calc` — when
`len(contributing) < 2` the goodness-of-fit is undefined; either reject at
validation with a 400, or set `p_value`/`is_null` to JSON `null`.
Independently, configure the Flask JSON provider with `allow_nan=False` so an
undefined statistic can never serialise as `NaN`. Add a regression test for
`dist=5:1.0` on both v1 and v2, then make production match it.

## Low severity

### F12 — Distribution category count is unbounded; O(n²) CDF precompute enables CPU-heavy requests

Category: security · Files: `service.py`, `domain/core.py`,
`blueprints/api.py` · Effort: M.

`MAX_NUMBERS` bounds only the draw count (service.py:89); nothing caps
`len(values)`. `parse_dist_pairs` (api.py:69) splits the whole `dist` string
with no length limit, and `RandomGenV1.calc_cdf` (core.py:196-198) is O(n²):
`sum(self._probabilities[: i + 1])` re-sums each iteration, run inside
`validate()` before the quantity check. Measured: `validate` alone ~17 s at
60k categories, ~43 s at 100k. On the deployed gunicorn path the default
`limit_request_line` (~4 KB) incidentally caps a GET to ~300 categories, but
that is an accidental backstop, not a deliberate control; the Werkzeug dev
server / in-process embedding has no such cap.

Fix: cap the category count in `parse_dist_pairs`/`validate_distribution` so
the bound lives in code; optionally set `MAX_CONTENT_LENGTH` and rewrite
`calc_cdf` to O(n) via `itertools.accumulate`. This is a category-count cap,
distinct from the out-of-scope request-rate limiting.

### F13 — No HTTP security response headers on HTML-serving routes

Category: security · Files: `app.py`, `observability.py`,
`blueprints/web.py` · Effort: S.

A repo-wide search for `X-Content-Type-Options` / `X-Frame-Options` /
`Content-Security-Policy` found matches only under `docs/`, never in
`src/randomgen/`. `create_app()` registers no `after_request` header hook;
`observability.py`'s `after_request` only logs. The home page and `/docs`
render full HTML. Low real-world impact for a stateless demo with no
auth/session, but a defense-in-depth gap the project's own `security.md`
calls for. Fix: add a small `after_request` hook setting `nosniff`,
`X-Frame-Options: DENY` (or CSP `frame-ancestors`), and a basic CSP. Defer
HSTS to the TLS platform per §3.3.

### F22 — No dependency vulnerability audit (pip-audit / SCA) in CI

Category: security · Files: `.github/workflows/ci.yml`,
`.github/dependabot.yml` · Effort: S.

`ci.yml` has six gates but none runs SCA; a grep for
`pip-audit|trivy|grype|syft|sbom` in `.github/` finds nothing. The vendored
`devsecops.md` states SCA must run on every deployment. Dependabot raises
bump PRs but does not fail a build on a known-vulnerable pinned dependency;
CodeQL scans first-party source only. Fix: add a one-step `sca` job running
`pip-audit`, wired into the `gate` fan-in.

### F23 — Published Docker image is never vulnerability-scanned; Dockerfile not IaC-scanned

Category: security · Files: `.github/workflows/cd.yml`, `Dockerfile` ·
Effort: S.

`cd.yml` builds, pushes, and deploys the image with no scan anywhere.
`devsecops.md` requires Dockerfiles to be IaC-scanned in CI. The image is the
project's primary distribution artifact (what interviewers/clients pull), so
OS-package CVEs in the pinned base accumulate unseen. Fix: add a `trivy image`
(or grype) step after the push, failing on HIGH/CRITICAL (advisory exit-code
acceptable for unpatchable upstream-base CVEs).

### F25 — Digest-pinned base image is ~2 years stale and will accumulate CVEs

Category: security · Files: `Dockerfile`, `.github/dependabot.yml` ·
Effort: S.

`Dockerfile:2` pins `python:3.12.2-alpine3.19@sha256:c7eb5c92...`, a tag last
updated 2024-03-25. Acknowledged as risk R5 in arc42 §11. `dependabot.yml`
declares only pip and github-actions — no docker ecosystem, so the base is
unmonitored. The pinning approach is correct; only the cadence is the gap.
Fix: bump to the latest `python:3.12.x-alpine`, refresh the digest, and add a
docker ecosystem to Dependabot.

### F2 — Out-of-domain observations inflate the denominator but are dropped from the statistic

Category: correctness · File: `domain/hypothesis.py` · Effort: S.

When `set_expected_numbers` defines the domain, `self._total` counts every
observation (hypothesis.py:272), but the chi-square sum iterates only over the
declared domain (hypothesis.py:283). `observed=[1,1,9,9]`,
`expected_numbers=[1,2]` gives `chi_square=2.0` with `_total=4` — the two 9s
inflate the expected counts yet never appear in any term. Unreachable via the
service (V1/V2 only emit values from `numbers`, and `set_expected_numbers`
always matches), so this is a latent hazard for direct/library reuse. Fix:
validate observed ⊆ expected_numbers, or compute `_total` from the
contributing categories only.

### F3 — `ChiSquareTest.calc` does not validate that probabilities sum to 1, allowing negative df

Category: correctness · File: `domain/hypothesis.py` · Effort: S.

`validate()` (hypothesis.py:244-254) checks types and emptiness but never the
probability sum. With all-zero probabilities the contributing list is empty,
so `df = -1` and `p_value` is `NaN`. The service path is protected because
`validate_distribution` enforces `round(sum,3)==1` first. Fix: add a
probability-sum (and non-negativity) check to
`validate_expected_probabilities` so the class enforces its own
preconditions.

### F4 — `RandomGenProbabilityNegativeError` is defined but never raised in the domain

Category: correctness · Files: `domain/core.py`, `domain/errors.py`,
`service.py` · Effort: S.

The dedicated error (errors.py:49-55) is raised only by
`service.validate_distribution` (service.py:61). In the domain,
`RandomGenABC.validate_probabilities` folds the negative check into the
generic type branch (core.py:125-127), so a negative weight validated through
the generator surfaces `RandomGenTypeError` ("must be a list of numbers")
rather than the specific message. Same input, different classification per
layer. Fix: raise `RandomGenProbabilityNegativeError` for the
`any(probability < 0)` case in `core.py`.

### F14 — Performance test bound (500 ms) contradicts its name and docstring (50 ms / 25 ms)

Category: correctness · File: `tests/test_core.py` · Effort: S.

`test_time` (test_core.py:351-369) has docstring "below 50ms", inline comment
"less than 25 msec", and asserts `delta < 50e7` = 500,000,000 ns = 500 ms —
10–20× looser than stated, and the docstring/comment contradict each other. A
real 10× regression would still pass. Fix: pick the intended budget and make
assertion, name, docstring, and comment agree (e.g. `delta < 50e6`), or
relabel honestly as a smoke check.

### F7 — Three unused `__main__` demo blocks are dead code

Category: clarity · Files: `domain/core.py`, `domain/histogram.py`,
`domain/hypothesis.py` · Effort: S.

Three `if __name__ == '__main__'` demo blocks (core.py:251-265,
histogram.py:114-125, hypothesis.py:344-361) are genuinely dead: no entry
point, no `python -m`, no test invokes them. They ship in the wheel unexecuted
and contribute to the uncovered-line baseline. Caveat from verification: they
are not the only uncovered lines (abstract-method bodies and a few others
count too), and `core.py`'s `import random` (line 1) is used by production
(`random.choices` at line 244) and must not be removed. `healthcheck.py`'s
`__main__` is a real Docker entrypoint to keep. Fix: remove the three blocks
(relocate any useful demo to `scripts/`).

### F8 — `Histogram._probabilities` is initialised and documented but never used

Category: clarity · File: `domain/histogram.py` · Effort: S.

`histogram.py:22` sets `self._probabilities = ()` and the docstring documents
it, but it is never read or reassigned (computed proportions go into the dict
itself at line 105). Misleading dead state. Fix: remove the attribute and its
docstring line, or populate/use it.

### F18 — Spec documents only 200/400 for `/api` routes, but the service returns 404/405/500

Category: clarity · Files: `openapi.yaml`, `app.py` · Effort: S.

`openapi.yaml` (lines 36-41, 59-64) documents only 200 and 400, but
`handle_error` preserves Werkzeug HTTP errors and returns 500 for unexpected
failures. Live: `POST .../randomgen` → 405, `GET /api/v3/randomgen` → 404,
both using the documented Error envelope. The behaviour is correct; only the
contract is silent. Fix: add a shared 4XX/5XX response referencing the Error
schema (or a `components/responses` default), plus a drift-guard test.

### F28 — arc42 §3.3 scope and §4.2 FR-mapping omit FR08 (`/info`)

Category: clarity · Files: `docs/arc42/03-context-and-scope.md`,
`docs/arc42/04-solution-strategy.md`, `docs/arc42/01-introduction-and-goals.md`
· Effort: S.

§1.1 defines FR08 and the endpoint is implemented (web.py:105), but §3.3 "In
scope" still reads "FR01–FR07" and the §4.2 table stops at FR07. Internal
inconsistency. Fix: update §3.3 to "FR01–FR08" and add an FR08 row to §4.2.

### F29 — `CLAUDE.md` src tree omits the `healthcheck.py` module

Category: clarity · Files: `CLAUDE.md`, `healthcheck.py`, `Dockerfile` ·
Effort: S.

The `CLAUDE.md` project-structure tree (lines 50-66) lists every other
top-level module under `src/randomgen/` but not `healthcheck.py`, which ships
in the package, has unit tests, and backs the Docker liveness probe. Fix: add
a `healthcheck.py` line to the tree.

### F6 — List-of-numbers validation logic is near-duplicated five times across four files

Category: clarity · Files: `domain/core.py`, `domain/histogram.py`,
`domain/hypothesis.py`, `service.py` · Effort: M.

The same "None / not iterable / is dict / not all int|float →
`RandomGenTypeError`, then empty → `RandomGenEmptyError`" block appears at
core.py:77-87, histogram.py:61-71, hypothesis.py:174-184, and
hypothesis.py:230-240; `service.validate_distribution` (service.py:51-66)
re-implements type/empty/length/sum. The copies have drifted: core.py orders
the iterable check before the dict check; histogram.py and hypothesis.py order
the dict check first. Fix: extract a single `validate_number_iterable(values)`
helper in `domain/` and call it everywhere; keep behaviour identical to
preserve the `/api/v1` and `/api/v2` contract.

### F15 — The 85% coverage floor runs only in CI, not the local gate, and has no dead-code exclusions

Category: conventions · Files: `pyproject.toml`, `.github/workflows/ci.yml` ·
Effort: S.

`pyproject` `addopts` carries no `--cov`/`--cov-fail-under` and there is no
`[tool.coverage]` section; the floor lives only in `ci.yml:73`. `CLAUDE.md`
tells developers to run plain `pytest` before every commit, which produces no
coverage. No `omit`/`exclude_lines` for `__main__` blocks or abstract bodies,
so the reported 92% is dragged down by intentionally-dead code. Fix: add
`[tool.coverage.report] exclude_lines` for `if __name__ == .__main__.:` and
`raise NotImplementedError`; optionally surface coverage in local runs.

### F27 — Inconsistent `~=` pin precision across runtime dependencies

Category: conventions · File: `pyproject.toml` · Effort: S.

`scipy~=1.17.1` and `flask~=3.1.3` lock the minor (allow patch), whereas
`gunicorn~=26.0` and `pyyaml~=6.0` float the whole minor range.
`scripts/requirements.txt` also duplicates the flask pin and pins requests at
a different precision than the extras. Reproducibility guarantees differ per
dependency. Fix: pick one precision for all four runtime pins; align/dedupe
`scripts/requirements.txt`.

## Informational

### F5 — V1 inverse-CDF can return a leading zero-probability category when `random()` yields exactly 0.0

`next_num` uses `if rand <= cum_prob` (core.py:221). For
`probabilities=[0.0,0.5,0.5]` the CDF is `[0.0,0.5,1.0]`; when `rand==0.0` the
first iteration returns `_numbers[0]` (probability 0). Probability ~2⁻⁵³ per
draw; V2 is unaffected. Not worth a code change — recorded for completeness;
document leading zero-weight categories as unsupported if ever addressed.

### F16 — `Histogram.validate()` wrapper is never exercised

Coverage shows histogram.py:83-84 uncovered — the body of
`Histogram.validate()`. Every histogram test calls `validate_numbers()`
directly. Real production code, not a `__main__` block. Fix: add a one-line
test or drop the unused wrapper.

### F21 — Drift guard pins Numbers default/maximum but not minimum

`openapi.yaml` declares `minimum: 1` (line 104), enforced by `if quantity <= 0`
(service.py:85-86). `test_spec_pins_live_limits` asserts `default` and
`maximum` but not `minimum`, and the minimum has no named constant like
`MAX_NUMBERS`. Latent guard gap. Fix: add `MIN_NUMBERS = 1` and assert
`numbers['minimum'] == MIN_NUMBERS`.

### F9 — `home()` duplicates four values and ships two unread config keys

`home()` (web.py:64-76) passes version/default_quantity/default_dist/
max_numbers as snake_case kwargs and repeats them in a camelCase `config`
dict; `app.js` (lines 294-295) reads only `cfg.defaultQuantity` and
`cfg.defaultDist`, so `config['maxNumbers']` and `config['version']` are dead
payload. Fix: derive the camelCase config from one source, or drop the unused
keys.

### F17 — Negative tests wrap setter + validator in one `pytest.raises`

E.g. test_core.py:35-37 wraps `set_numbers(None); validate()` in a single
`pytest.raises`. Today only `validate()` raises, but the block would also pass
if a future change made the setter raise the same type, masking which call
enforces the rule. Fix: scope `pytest.raises` to the single validating
statement where practical.

### F19 — Response and metadata schemas omit `required`

Only the Error schema declares `required` (openapi.yaml:148-149).
RandomGenResponse, Quality, ChiSquareTest, Info, and Health omit it though all
fields are always present, so generated client models mark always-present
fields as optional. Fix: add `required` arrays matching the guaranteed
payload.

### F20 — No `servers` block in the OpenAPI document

The spec has no top-level `servers:` key, so ReDoc "Try it" and generated
clients have no declared base URL. Declaring a server URL is documentation,
not TLS termination, so it does not breach §3.3. Fix: add a `servers` list
(the Render demo URL and/or a relative `/`).

### F10 — No type annotations on `domain/` methods or blueprint helper signatures

A grep for `->` and parameter hints across `domain/` returns no matches; e.g.
`set_numbers(self, values)` (core.py:54), `generate(...)` (service.py:129),
`parse_dist_pairs(raw)` (api.py:45). `CLAUDE.md` §2.2 asks to annotate public
signatures; `healthcheck.py`/`openapi.py` already do. Fix: annotate the
straightforward signatures where it does not fight scipy's missing stubs.

### F11 — `web` blueprint imports presentation constants from the application layer

`blueprints/web.py` (lines 12-17) imports `DEFAULT_NUMBERS`,
`DEFAULT_PROBABILITIES`, `DEFAULT_QUANTITY`, `MAX_NUMBERS` from `service.py`. A
documented AD-22 trade-off (web → service is the correct inward direction
under AD-26), so optional. Fix (optional): move shared defaults/`MAX_NUMBERS`
to a small constants module, or leave as-is.

### F26 — gunicorn sets no explicit request timeout or worker recycling

`gunicorn.conf.py` defines only `bind` and `workers`; no `timeout`,
`graceful_timeout`, `max_requests`, or `worker_class`. Not an exploitable
threat — `MAX_NUMBERS=10000` caps draws and generation is sub-second — so this
is config hygiene, not a timeout risk. Fix (optional): set an explicit
`timeout` and `max_requests`/`max_requests_jitter` to document intent.

### F24 — No SBOM generated on release

`devsecops.md` states an SBOM must be generated per release; `cd.yml` produces
none. For a solo demo the operational benefit is small, but it is a stated
MUST in the project's own framework. Fix: add an `anchore/sbom-action`/`syft`
step emitting an SPDX/CycloneDX SBOM as a release asset, or record an ADR
waiver so the deviation is intentional.

## Strengths

- The `domain/` layer is genuinely framework-free (AD-26): a grep for
  flask/werkzeug/blueprints/service/app imports under `domain/` returns
  nothing. The Clean Architecture dependency rule is enforced in practice.
- `create_app()` (app.py:58-88) is a textbook composition root with correct
  ordering (config → logging hooks → blueprints → single Exception boundary
  last), and `API_VERSIONS` (versions.py:18-21) decouples version selection
  from the version-agnostic domain.
- V1 (inverse-CDF) and V2 (`random.choices`) demonstrably sample the same
  distribution (empirical proportions over 200k draws match), and the
  chi-square uses correct raw-count arithmetic with df = k−1 over the full
  declared domain.
- The OpenAPI 3.1 contract is a faithful design-first single source of truth,
  served verbatim at `/openapi.json`, matching live responses field-for-field,
  with drift guards (`test_openapi.py` pins limits/routes to live constants)
  and real v1/v2 parity enforced structurally by the shared blueprint factory.
- The centralized error handler (app.py:24-55) is a clean single API boundary:
  domain errors → 400 (static message), HTTPException preserved, everything
  else → generic 500 with full server-side logging and no internal leakage.
- Two-layer input validation fails loud (`quantity_from_query` raises rather
  than silently defaulting), and logs capture only static rejection reasons,
  never attacker-controlled payloads.
- CI/CD is mature for a solo demo: one-gate-per-job SRP with a correct
  `if: always()` fan-in that fails on any non-success, every action SHA-pinned
  with version comments, workflow-scope least-privilege permissions, weekly
  CodeQL cron, gitleaks on full history, guarded artifact promotion.

## Scope-excluded by design

These were considered and deliberately not recommended — arc42 §3.3 excludes
them. Recorded so the scope decision is transparent rather than silent.

| Idea | Why excluded |
|------|--------------|
| Authentication / authorization | §3.3 deliberate exclusion; stateless public demo with no user data or sessions. |
| API rate limiting / CORS | §3.3 deliberate exclusion. (F12's category-count cap is input validation in the project's own code, distinct from request-rate limiting, and stays in scope.) |
| Cryptographically secure RNG | §3.3 exclusion; Mersenne-Twister is appropriate for a goodness-of-fit demo, not an unpredictability use-case. |
| Persistence / database | §3.3 exclusion; CLAUDE.md §2.3 confirms pure compute via scipy, intentionally stateless. |
| HTTPS / TLS termination in the app | §3.3 designates TLS as the platform's job (plain HTTP by design). F13 recommends only app-owned headers and defers HSTS to the platform. |

## Appendix — method

- 10 dimension reviewers ran in parallel (correctness, code quality,
  architecture, API contract, security, tests, CI/CD, container, dependencies,
  docs), each reading the actual files (not summaries).
- Every finding was handed to an independent skeptic agent that re-read the
  cited `file:line` and tried to refute it; default-to-refuted on uncertainty.
  4 of 37 were dropped or recharacterised.
- The headline F1 was additionally reproduced by the human-in-the-loop via the
  Flask test client before being featured.
- Coverage (92%), determinism (207 pass with `-p no:randomly`), and the
  category-count timing were measured by running code, not asserted from
  reading.
