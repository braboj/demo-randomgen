# RandomGen — 360° Review Plan

- Date: 2026-06-24
- Subject: `randomgen` v0.19.0 (API contract 2.1.0), branch `main`
- Companion: [2026-06-24-360-review.md](2026-06-24-360-review.md) — the
  findings this plan produced.
- Status: plan of record. Captured to pair with the report so the method is
  auditable and repeatable, not just its results.

## Objective

Run a whole-project 360° assessment of v0.19.0 before moving toward v1.0:
grade the project across every engineering dimension, surface blind spots a
single-pass review would miss, and convert actionable findings into a triaged
GitHub backlog — without changing source (read-only assessment).

## Relationship to the base-360 template

This review adapts `templates/base/workflow/360.md`. That template frames a
360 as four stakeholder perspectives (Value, Quality, Viability, Discovery)
for a product with a user-facing surface. RandomGen is a stateless backend
demo whose "product" is its code, contract, and pipeline, so the four
stakeholder lenses collapse almost entirely into the engineer's. The plan
keeps the template's core mechanics — parallel role-isolated subagents, clean
per-agent context, letter grades, a persisted report — and re-projects the
single Quality perspective into ten code dimensions, each given the isolated
context the template prescribes.

| Template element | Kept | Adapted |
|---|---|---|
| Parallel role-isolated subagents | ✅ | One agent per dimension instead of per stakeholder |
| Clean per-agent context (no bleed) | ✅ | — |
| Letter grade per category + overall | ✅ | Overall = lowest dimension grade |
| Persisted report in `docs/` | ✅ | `docs/review/` rather than `docs/360-audit.md` |
| 4 stakeholder perspectives | — | Re-projected to 10 engineering dimensions |
| Marketing / SEO / social checklist | — | Out of frame for a backend demo (no public landing surface to discover) |

## Execution model

A three-stage fan-out: independent finders → adversarial verifiers →
synthesis. 48 agents total (10 finders + 37 per-finding skeptics + 1
synthesiser), with a human-in-the-loop live reproduction of the headline bug.

```
Parent (orchestrator)
├── Stage 1 — Find:   10 dimension reviewers in parallel  → 37 raw findings
├── Stage 2 — Verify: 1 skeptic per finding, in parallel  → 33 survive (4 dropped/recharacterised)
└── Stage 3 — Synthesise: dedup → 29 findings → grade, severity, issue map
```

Each finder starts from a clean context and reads the actual files at the
cited `file:line` — no summaries, no shared running list, no awareness of
other dimensions' findings. Isolation is structural (separate context
windows), which is what stops every lens from collapsing into one generic
pass.

## Scope guard

Every recommendation is reconciled against the documented scope boundary in
arc42 §3.3 before it is raised. Deliberate exclusions — auth, request-rate
limiting / CORS, crypto-grade RNG, persistence, TLS — are recorded under
"Scope-excluded by design" in the report rather than reported as gaps. A
finding that would only matter under a different scope is dropped here, not
verified. This applies the `scope-vs-audit` rule: a written boundary
(arc42 §3.3, set by ADR) outranks generic "demo value" pressure; the place to
widen scope is an ADR, not a review finding.

## Stage 2 — Adversarial verification

Each raw finding is handed to a fresh skeptic agent whose only job is to
refute it. The skeptic re-reads the cited `file:line`, defaults to refuted on
any uncertainty, and must reproduce the claim's basis from the code rather
than the finder's summary. Findings that survive are kept; the rest are
dropped or recharacterised. Of 37 raw findings, 33 survived and 4 were
dropped or downgraded. The headline correctness bug (F1) was additionally
reproduced by the human-in-the-loop through the Flask test client before being
featured — the strongest tier of evidence in the report.

Quantitative claims are measured by running code, never asserted from reading:
coverage (92%), determinism (207 tests pass under `-p no:randomly`), and the
category-count timing in F12 were all produced by execution.

## Stage 3 — Synthesis and output

The orchestrator merges the surviving findings:

1. Dedup overlapping reports across dimensions (33 → 29 reported findings).
2. Assign severity (High / Low / Info) and effort (S / M / L).
3. Grade each dimension A–F; the overall grade is the lowest dimension grade
   (a project is only as strong as its weakest perspective).
4. Group findings into atomic GitHub issues and record the finding→issue map.
